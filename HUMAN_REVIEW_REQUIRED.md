# Human Review Required

Date: 2026-05-31

This file lists decisions that should be made by a human before future agents perform implementation refactors or science-changing optimization.

## Ambiguous Classifications

- `modular-pipeline/analysis/Higgs-to-diphoton.json` appears to duplicate `modular-pipeline/analysis/analysis.summary.json`, but skill/runtime examples reference the former while README examples reference the latter.
- `modular-pipeline/analysis/regions.yaml` is generated by `write_regions_yaml`, but it exists in the source tree.
- `modular-pipeline/setup.py` may be a legacy packaging shim now that `pyproject.toml` is authoritative.
- `.TinyTeX/` appears to be a local toolchain under the workspace rather than analysis source.
- `analysis_documentation_assets/generate_assets.py` could be a report service, documentation utility, or exploratory script.

## Missing Physics-Policy Decisions

- Which analysis-summary path is canonical?
- Which values in `DEFAULT_RUNTIME` are immutable physics policy versus tunable optimization configuration?
- What are the allowed bounds, if any, for changing photon, jet, category, mass-window, and sideband definitions?
- Who can approve `--unblind-observed-significance`, and how should that approval be recorded?
- Is the current placeholder systematics model acceptable for optimization ranking, publication-style reporting, or only software smoke tests?
- Should HHXYY quickFit be the central expected-significance backend whenever available, or a cross-check of local RooFit output?

## Proposed Service Boundaries Needing Approval

- `summary_normalizer`: validate and materialize effective config.
- `sample_registry`: sample discovery, classification, nominal/alternative policy, normalization table.
- `event_processor_histogram_builder`: object building, selections, event weights, cutflows, templates.
- `observable_materializer`: future validated producer graph for input branches, expressions, Python plugins, and learned-score columns. The current Section 8 shared builder is only an internal compatibility seam.
- `category_router`: future generic ordered first-match router. It must prove compatibility with five-category routing and Section 8 assigned/unassigned/blocked tri-state behavior before promotion.
- `learned_observable_service`: future model-training and score-materialization interface for BDT and NN score columns. It must keep model artifacts versioned separately from score columns.
- `fit_builder`: RooFit workspace/model construction and measurement fits.
- `significance_calculator`: expected and observed significance with blinding enforcement.
- `background_selector_bridge`: optional official HHXYY background-selector integration.
- `plot_report_builder`: blinded plotting, reports, and validation artifacts.
- `artifact_tracker`: manifests, state, readiness, and provenance.

## Duplicated Or Overlapping Implementations

- `analysis/pipeline.py` and `modular_pipeline/components.py` both encode the stage order. The current design intentionally keeps the former as canonical behavior and the latter as orchestration, but future edits must avoid drift.
- `analysis/analysis.summary.json` and `analysis/Higgs-to-diphoton.json` appear duplicated.
- Root `ANALYSIS_DOCUMENTATION.*`, `analysis_documentation_assets/*`, and generated pipeline report artifacts overlap as documentation sources.
- Local PyROOT fit/significance and HHXYY/quickFit backend outputs overlap; central-versus-cross-check roles must remain explicit.
- Local background selection and official HHXYY background selector bridge overlap by design; substitution must be opt-in and reviewed.

## Likely Refactoring Risks

- Moving runtime defaults out of Python could accidentally change effective configuration hashes or defaults.
- Refactoring event weights, object definitions, or category assignment could silently alter yields.
- Changing sample classification tokens could alter nominal signal/background selection.
- Changing fit parameter policies can produce numerically successful but scientifically wrong fits.
- Treating masked modular runs as resumed full runs can overstate validation.
- Mixing action services with verifier decisions can allow successful completion to masquerade as correctness.

## Proposed Invariants For Approval

- Data event weights remain 1.0.
- MC event weights retain the declared normalization formula.
- Observed significance remains blinded by default.
- Signal-region categories are exclusive unless explicitly declared otherwise.
- Histogram outputs record bin edges, weighted counts, `sumw2`, yields, and entries.
- Every optimization run preserves exact config snapshots, service versions, input sample provenance, and git/working-tree state.
- Physics-policy changes and backward-incompatible service changes require explicit human approval.

## Backward-Compatibility Concerns

- CLI entrypoint names in `pyproject.toml` should remain stable.
- Existing JSON output field meanings should remain stable until schemas and migrations exist.
- Existing summary JSON files should continue to run after config extraction.
- Existing generated output directories should remain inspectable by `modular-pipeline inspect`.
- Existing tests only cover a small slice of behavior; broader compatibility tests should be added before service refactors.
